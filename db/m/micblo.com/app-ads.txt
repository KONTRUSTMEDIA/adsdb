<img src="/2022/06/14/%E7%94%A8JavaScript%E8%A7%A3%E6%9E%90Android-XML-axml/parse-apk-manifest.jpg" class title="解析AndroidManifest.xml的小工具">

<p>其中，整个项目最复杂的地方就是 <code>axml</code> 转换 <code>xml</code> 的实现。<br>因为 <code>AndroidManifest.xml</code> 并非是一个真正的 <code>XML</code>，而是一个 <code>axml</code> 格式的文件，需要借助特殊的工具解析、转换到 <code>XML</code>。</p>
<p>在 npm 上找了半天，没找到符合我要求的库。</p>
<blockquote>
<p>NPM 上 <code>axml</code> 那个库和 Android 的 XML 完全没有关系！</p>
</blockquote>
<p>没办法，只能自己动手实现一下了！</p>
<p>借鉴了 <a target="_blank" rel="noopener" href="https://github.com/xgouchet/AXML">https://github.com/xgouchet/AXML</a> 的源代码，我迁移了一个 JavaScript 版本的 <a target="_blank" rel="noopener" href="https://www.npmjs.com/package/axml2xml">axml2xml</a>。伸手党可以直接用下面的命令添加这个库：</p>
<figure class="highlight bash"><table><tr><td class="gutter"><pre><span class="line">1</span><br></pre></td><td class="code"><pre><span class="line">npm install axml2xml --save</span><br></pre></td></tr></table></figure>

<p>崩溃堆栈类似下面：</p>

<p>通过 Bugly 上报的数据发现，这个 <code>SEGV_ACCERR</code> 的崩溃（无访问权限）发生在读取 <code>libc.so</code> 的代码段(<code>.text</code>)内存的时候。<br>观察其他信息发现，<code>libc.so</code>的代码段权限只剩下 <code>--xp</code>。<br>怪不得我们一访问就崩溃，<strong>是因为代码段没有可读权限了！</strong><br>同样的问题也发生在其他系统库上。</p>
<p>当时的解决方案是，通过读取 <code>/proc/self/maps</code> 避开系统库无可读权限的用户。</p>
<p>直到我不小心看到了<a target="_blank" rel="noopener" href="https://source.android.google.cn/devices/tech/debug/native-crash?hl=zh-cn#xom">这个文档</a>……</p>
<blockquote>
<p><strong>只执行内存违规（仅限 Android 10）</strong></p>
<p>仅对于 Android 10 中的 arm64，二进制文件和库的可执行部分会映射到只执行（不可读取）内存，作为防范代码重用攻击的一种安全强化技术。此缓解方法与其他缓解方法产生了不良互动，后已被移除。</p>
<p>将代码设为不可读会导致故意或意外读入已标记为只执行的内存段抛出 SIGSEGV，并且代码为 SEGV_ACCERR。这可能是因为错误、漏洞、混合了代码的数据（例如文字池）或故意进行的内存自省导致的。</p>
<p>编译器会假设代码和数据不是混合在一起的，但是手写程序集会导致出现问题。在许多情况下，只需将常量移动到 .data 部分，即可解决这些问题。如果可执行代码段绝对有必要进行代码内省，应首先调用 mprotect(2) 以将该代码标记为可读，然后在操作完成后重新将该代码标记为不可读。</p>
</blockquote>
<p>寻思这不是Android系统你在搞事情啊！<br>Android 官方提供的解决方案是访问之前先 <code>mprotect</code> 一下需要读的内存，给他们加上可读权限。</p>
<p>提供一个伪代码:</p>

<p>在2020的现在，当年的那些PowerPoint里面的Flash已经无法在Office 2016以及更高的版本播放了。更糟糕的是，你想直接导出文件也变得极度困难。</p>
<p>目前网上常见的招数是使用WinHex之类的软件直接硬导出 <code>.ppt</code> 内的文件。该方法虽然在一定程度上可行，但是过于硬核，实际操作难度非常大。</p>
<p>为了解决这个问题，我从另一个角度提取PPT中的Flash文件。</p>
<p>这种方法必须将PowerPoint文件转成 <code>.pptx</code> 格式。原因是 <code>.pptx</code> 格式的本质是一个 <code>.zip</code> 的压缩包，我们可以使用压缩工具解压里面的资源文件。</p>
<p>使用压缩工具打开 <code>.pptx</code> 文件后，可以在 <code>ppt/activeX</code> 文件夹找到PowerPoint中所有的 ActiveX 插件对应的原始的原始文件。其名称类似于 <code>activeX&#123;数字&#125;.bin</code>。</p>
<p>那如何识别这个 <code>.bin</code> 文件原本是什么文件呢？很简单，该文件后缀为 <code>.xml</code> 的资源配置文件里面就有答案。</p>
<figure class="highlight xml"><table><tr><td class="gutter"><pre><span class="line">1</span><br><span class="line">2</span><br></pre></td><td class="code"><pre><span class="line"><span class="meta">&lt;?xml version=<span class="string">&quot;1.0&quot;</span> encoding=<span class="string">&quot;UTF-8&quot;</span> standalone=<span class="string">&quot;no&quot;</span>?&gt;</span></span><br><span class="line"><span class="tag">&lt;<span class="name">ax:ocx</span> <span class="attr">ax:classid</span>=<span class="string">&quot;&#123;D27CDB6E-AE6D-11CF-96B8-444553540000&#125;&quot;</span> <span class="attr">ax:persistence</span>=<span class="string">&quot;persistStorage&quot;</span> <span class="attr">r:id</span>=<span class="string">&quot;rId1&quot;</span> <span class="attr">xmlns:ax</span>=<span class="string">&quot;http://schemas.microsoft.com/office/2006/activeX&quot;</span> <span class="attr">xmlns:r</span>=<span class="string">&quot;http://schemas.openxmlformats.org/officeDocument/2006/relationships&quot;</span>/&gt;</span></span><br></pre></td></tr></table></figure>
