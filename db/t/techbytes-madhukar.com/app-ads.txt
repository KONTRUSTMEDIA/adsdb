<h3 class='post-title entry-title' itemprop='name'>
<a href='https://askmadhukar.blogspot.com/2025/11/using-sftp-on-macos.html'> Using SFTP on macOS</a>
</h3>
<div class='post-header'>
<div class='post-header-line-1'></div>
</div>
<div class='post-body entry-content' id='post-body-5555098945212467216' itemprop='articleBody'>
<br /><br />Method 1 SFTP (Secure File Transfer Protocol) is a secure way to transfer files between systems. macOS provides built-in support for SFTP through the terminal, and you can also use third-party tools for a graphical interface.<br /><br /><br />Method 1: Using Terminal (Built-in SFTP Client)<br /><br /><br />Open Terminal Launch the Terminal app from Applications &gt; Utilities or by searching for it in Spotlight.<br /><br /><br />Connect to the Server Use the following command to initiate an SFTP session: sftp username@hostname Replace username with your server username and hostname with the server's IP address or domain.<br /><br /><br />Authenticate Enter your password when prompted.<br /><br /><br />Transfer Files Use these commands to manage files: Upload a file: put /path/to/local/file /path/to/remote/destination Download a file: get /path/to/remote/file /path/to/local/destination List files: ls Exit SFTP session: exit<br /><br /><br />Method 2: Using FileZilla (Graphical Interface)<br /><br />Download and Install FileZilla Download FileZilla Client for macOS from FileZilla's official site. Ensure you select the version compatible with your macOS version.<br /><br /><br />Launch FileZilla Open the application after installation.<br /><br /><br />Set Up Connection Enter the following details in the top bar: Host: sftp://hostname Username: Your server username. Password: Your server password. Port: Default is 22.<br /><br /><br />Transfer Files Drag and drop files between the local and remote panes to upload or download files.<br /><br /><br />Best Practices<br /><br /><br />Always use strong passwords or SSH keys for authentication.<br /><br /><br />Verify server authenticity to avoid man-in-the-middle attacks.<br /><br /><br />Use third-party tools like FileZilla for ease of use if you're not comfortable with command-line operations.

</h3>
<div class='post-header'>
<div class='post-header-line-1'></div>
</div>
<div class='post-body entry-content' id='post-body-7679979197375899501' itemprop='articleBody'>
<br />Pasting from the Windows clipboard into a PuTTY vi or vim terminal can sometimes be problematic, particularly with newer vim versions that have mouse mode enabled by default. This can lead to unintended visual mode activation or incorrect pasting.<br /><br /><br />Here are the primary methods to paste from the clipboard into a PuTTY vi/vimsession:<br />Shift + Right-Click: <br />Shift + Insert: <br />Middle Mouse Button (Wheel Click): <br />Disable vim Mouse Mode (if applicable): <br />vim Paste Mode: <br />Note: Ensure vi or vim is in insert mode (by pressing i) before attempting to paste if you want the content to be inserted directly into the file. Otherwise, pasting might be interpreted as vim commands.<br />For larger pastes, vim's paste mode can be helpful. <br /><br />Enter insert mode in vim, then type :set paste and press Enter. <br /><br />Paste your content using one of the methods above, then type :set nopaste and press Enter to exit paste mode.<br /><br />If you are experiencing issues due to vim's mouse mode, you can disable it for the current session by typing :set mouse= and pressing Enter while in vim. <br /><br />To make this change permanent, you can add set mouse= to your ~/.vimrc file. <br /><br />If your mouse has a middle button (often the scroll wheel), clicking it can paste the content.<br /><br />Shift + Insert: <br />This keyboard shortcut can also be used to paste the clipboard content.<br /><br />This is often the most reliable method for pasting in PuTTY, especially when vim's mouse mode is active. Hold down the Shift key and then right-click in the PuTTY window where you want to paste.<br /><br />

          <div class="date-posts">
        
<div class='post-outer'>
<div class='post hentry uncustomized-post-template' itemprop='blogPost' itemscope='itemscope' itemtype='http://schema.org/BlogPosting'>

<h3 class='post-title entry-title' itemprop='name'>
<a href='https://askmadhukar.blogspot.com/2025/10/what-caused-aws-outage-that-broke.html'>What caused the AWS outage that broke the internet on Oct 21st 2025</a>
</h3>
<div class='post-header'>
<div class='post-header-line-1'></div>
</div>
<div class='post-body entry-content' id='post-body-6625129011824947584' itemprop='articleBody'>
<a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=AWS+outage+2025&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">AWS outage 2025</a>: More than 1,000 services were impacted by the outage, including popular platforms like <a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=WhatsApp&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">WhatsApp</a>, <a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=Snapchat&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">Snapchat</a> and <a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=Reddit&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">Reddit</a>, which rely on AWS services, along with financial institutions like the <a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=British+government+tax+services+AWS+outage&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">British government&#8217;s tax services</a> and entertainment services<div><span style="font-family: &quot;Droid Serif&quot;, serif; font-size: 16px;">In India, the impact of the outage was most pronounced in the aviation sector, with hundreds of flights delayed and several cancelled as airline operators found their systems inoperational and had to switch to manual processes. At least ten banks and NBFCs had &#8220;minor disruptions&#8221;, which have either been resolved or are being resolved, the <a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=Reserve+Bank+of+India&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">Reserve Bank of India</a> said at the time.</span></div><div><span style="font-family: &quot;Droid Serif&quot;, serif; font-size: 16px;"><br /></span></div><div><u><b>Here is a technical explanation for the AWS Outage due to a <a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=what+is+DNS+failure&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">DNS failure</a>.<br /></b></u><br />AWS's own internal control plane is built on top of <a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=AWS+DynamoDB&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">DynamoDB</a>. It's a hidden dependency. When AWS's internal services couldn't find the IP for DynamoDB, the entire management layer collapsed.<br /><br />Stage 1: DNS Fails. The internal DNS servers for <a href="http://dynamodb.us-east-1.amazonaws.com/">dynamodb.us-east-1.amazonaws.com</a> stopped working.<br /><br />Stage 2: Control Plane Fails. AWS's own services that depend on DynamoDB immediately broke. This included:<br /><br /><a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=AWS+IAM&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">IAM</a> (for authentication and session state)<br /><br /><br />The <a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=AWS+EC2+instance+launch+subsystem&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">EC2 instance launch subsystem</a> (which uses DynamoDB for metadata)<br /><br /><br /><a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=AWS+Network+Load+Balancer&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">Network Load Balancer</a> (NLB) health checks (which, it turns out, write their health state to a DynamoDB table)<br /><br />Stage 3: Circular Dependency. This is the crazy part. When the <a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=AWS+NLB+health+checks&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">NLB health checks</a> failed (because they couldn't write to DynamoDB), it caused more network connectivity issues, which in turn impacted the (already struggling) DynamoDB service itself. It created a vicious feedback loop.<br /><br />Why it lasted 15+ hours (The <a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=define+UDP&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">UDP</a> problem)<br /><br />Fixing the DNS issue only took a couple of hours. The reason the recovery took so long was twofold:<br /><br />The Retry Storm: DNS queries use UDP, which is a stateless, "fire and forget" protocol. When the DNS queries failed, millions of clients (SDKs, <a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=AWS+Lambda+functions&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">Lambda functions</a>, other AWS services) didn't get an immediate "connection refused" (like with <a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=define+TCP&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">TCP</a>). They just timed out after 5+ seconds and then retried. This created a "<a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=retry+storm&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">retry storm</a>" (or <a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=thundering+herd&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">thundering herd</a>) of millions of requests that hammered the DNS servers and caches, preventing them from recovering even after the initial fix was in.<br /><br />The <a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=AWS+Global+Control+Plane&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">Global Control Plane</a>: Many of AWS's core control plane services (like IAM) are centralized in us-east-1. Even if your app was running in <a data-preview="" href="https://www.google.com/search?ved=1t:260882&amp;q=AWS+eu-west-1+region&amp;bbid=6626489648402552216&amp;bpid=6625129011824947584" target="_blank">eu-west-1</a>, if it needed to authenticate or launch an instance, that control plane operation was routed through us-east-1 and failed.</div>
